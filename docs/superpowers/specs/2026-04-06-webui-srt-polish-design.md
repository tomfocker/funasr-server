# WebUI 与 SRT 打磨设计

## 背景

当前 Docker 版 `Fun-ASR-Nano` 服务已经能稳定完成转录，但还有三个明显短板：

1. WebUI 结果区过于原始，缺少可直接交付用户的下载能力。
2. `SRT` 直接使用字符级时间戳渲染，容易出现“一字一条字幕”的碎片化结果。
3. 其他 Docker 应用虽然已经可以通过 HTTP 调用服务，但缺少清晰的对接说明和面向容器部署的显式配置表达。

本轮目标是在不扩大模型范围的前提下，把现有服务打磨成更适合实际使用的单模型转录面板。

## 目标

- 保持 `Fun-ASR-Nano` 单模型方案不变。
- WebUI 提供更清晰的结果展示、摘要信息和多格式下载。
- SRT 从字符级时间戳聚合为句级/短句级字幕段，减少碎片化。
- 保持现有 HTTP API 可用，并补充适合容器互调的文档和配置。
- 在实现功能的同时清理重复逻辑，降低后续维护成本。

## 约束

- 不引入数据库或持久化任务系统。
- 不在服务端保存“上一次转录结果”的会话状态，避免额外生命周期复杂度。
- 兼容现有 `/v1/audio/transcriptions` 与 `/api/transcriptions`。
- 继续优先支持浏览器上传文件与容器内 HTTP 调用两个场景。

## 方案比较

### 方案 A：只改 SRT 渲染，其他保持不变

优点：

- 改动最小。
- 对当前 API 影响最小。

缺点：

- WebUI 仍然原始，下载能力缺失。
- 用户看不到聚合后的字幕结构，只能看到最终 SRT 文本。
- 对容器集成体验提升有限。

### 方案 B：保留字符级原始段，同时新增句级字幕段，WebUI 前端完成下载

优点：

- 原始字符级时间戳仍可保留，兼容现有调试和后续扩展。
- 新增 `subtitle_segments` 专门服务 SRT、预览和下载，职责清晰。
- 下载使用前端 Blob 即可完成，不需要服务端存储会话状态。
- 改动集中在格式化和界面层，整体风险低。

缺点：

- `verbose_json` 结构会增加字段，需要对应更新测试和文档。

### 方案 C：服务端额外生成下载文件并暴露独立下载端点

优点：

- 浏览器下载链接更传统。
- 理论上便于后续扩展归档功能。

缺点：

- 需要管理临时文件与清理策略。
- 增加状态和边界条件，不符合当前轻量服务目标。

## 推荐方案

采用方案 B。

核心思路：

- 在格式化层新增字幕聚合逻辑，将字符级时间戳转换为句级 `subtitle_segments`。
- `TranscriptionOutput` 同时返回：
  - `text`
  - `segments`：原始字符级/细粒度段
  - `subtitle_segments`：适合用户阅读与导出的聚合字幕段
  - `srt`
  - `model`
  - `timings`
- WebUI 以 `verbose_json` 为主展示结果，并在前端提供：
  - 文本下载
  - JSON 下载
  - SRT 下载
  - 字幕段预览
- 容器互调继续走现有 HTTP 端口，在 `docker-compose.yml` 中增加 `expose`，并在文档中明确：
  - 宿主机访问方式
  - 同一 Compose 网络下其他容器访问方式
  - 健康检查与接口示例

## 架构与职责

### 格式化层

文件：

- `capsweb/formatters.py`
- 新增辅助模块（若需要）

职责：

- 统一规范化原始时间戳片段。
- 将原始片段聚合成适合阅读的字幕段。
- 基于聚合字幕段渲染 SRT。

### 结果契约层

文件：

- `capsweb/contracts.py`

职责：

- 扩充 `TranscriptionOutput`，显式区分原始段与字幕段。

### 应用层

文件：

- `capsweb/app.py`
- `capsweb/templates/index.html`

职责：

- 保持现有 API 接口兼容。
- WebUI 默认走 `verbose_json`，渲染摘要、文本、字幕段和下载按钮。
- 对错误、空结果、下载文件名做统一处理。

### 文档与部署层

文件：

- `docker-compose.yml`
- `docs/docker-webui.md`
- `readme.md`

职责：

- 明确端口暴露与容器互调方式。
- 说明下载能力、返回结构与推荐调用姿势。

## 字幕聚合规则

聚合以“尽量自然阅读”为目标，规则按优先级叠加：

1. 遇到句末/停顿型标点优先断句。
2. 字数达到上限时断句，避免单条过长。
3. 当前片段与下一片段间隔超过阈值时断句。
4. 超长停顿无条件断句。
5. 生成字幕文本时去掉尾部冗余标点，但保留正文自然标点。

聚合输出每条字幕段至少包含：

- `id`
- `start`
- `end`
- `text`

## 测试策略

- 为格式化层新增回归测试：
  - 字符级时间戳能聚合成短句字幕。
  - 长停顿会触发断句。
  - SRT 基于聚合段而非字符段生成。
- 更新 API 测试：
  - `verbose_json` 包含 `subtitle_segments`
  - `srt` 输出来自聚合后的字幕段
- 保留现有健康检查与基础接口测试。

## 风险与处理

- 如果上游输出的字符时间戳极不稳定，聚合可能偶尔切分得不够理想。
  - 处理：把聚合逻辑集中在单一模块，后续可以继续调参数。
- 如果前端直接下载大 JSON 结果，浏览器内存会短暂持有 Blob。
  - 处理：当前转录结果规模很小，可接受；下载后立即释放 URL。
- `verbose_json` 增加字段可能影响极少数硬编码消费者。
  - 处理：只新增字段，不移除已有字段，保持向后兼容。
